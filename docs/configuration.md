# Configuration

Phase 2 populated local YAML configuration files for deterministic triage logic.
Phase 4 executes severity rules in `config/rules.yaml`. Phase 5 executes local
static MITRE mappings in `config/mitre_mapping.yaml`. Phase 6 selects defensive
triage recommendations from `config/triage_steps.yaml` and uses deterministic
templates for analyst-style explanations. Phase 7 groups alerts into incidents
using `config/grouping_rules.yaml`. Phase 8 adds deterministic redaction and
safe serialization. Phase 9 adds deterministic Markdown and JSON reporting.

The configuration files are local data only. They do not define network calls,
external AI API integrations, threat intelligence lookups, executable commands,
or live scanning behavior.

## Files

- `config/rules.yaml`: base severity values, modifier thresholds, sensitive
  destination ports, privileged synthetic usernames, known synthetic bad IPs,
  approved marker regex names, deterministic modifier rules, and override rules.
- `config/mitre_mapping.yaml`: static local educational mappings between
  synthetic event types and MITRE ATT&CK technique examples.
- `config/triage_steps.yaml`: defensive analyst-style triage step text for
  synthetic alerts.
- `config/grouping_rules.yaml`: deterministic local incident correlation rules.

## Severity Levels

- `LOW`
- `MEDIUM`
- `HIGH`
- `CRITICAL`

## Safety Rules

Config values must remain deterministic and inspectable. YAML is treated as data
only, never as executable logic. Pipeline engines load these files from disk and
apply transparent rules with traceable decisions.

## Phase 4 Classification Rules

The classifier loads base severity from local YAML. Modifier rules are applied in
the order they appear in `config/rules.yaml`, and severity is capped at
`CRITICAL`.

Implemented Phase 4 modifiers:

- `count >= 100`: raise by one level.
- `count >= 500`: raise by two levels.
- Sensitive destination port: raise by one level.
- Privileged synthetic username: raise by one level.
- `internal_asset` tag: raise by one level.
- `honeypot` tag: raise by one level.
- Approved synthetic marker regex in `raw_message`: raise by one level without
  exposing the marker value in CLI output.

Implemented Phase 4 override:

- Known synthetic bad source IP: override final severity to `CRITICAL`.

No config file should contain API keys, real credentials, real tokens, real
customer data, public production IPs, real domains, external enrichment URLs, or
offensive-tool execution instructions.

## Phase 5 MITRE Mapping

`config/mitre_mapping.yaml` contains local static mappings from synthetic event
types to educational MITRE ATT&CK examples. The mapper supports multiple
techniques per event type and uses the `unknown` mapping as a documented fallback
for unmapped event types.

MITRE URLs are generated as static reference strings from technique IDs, such as
`https://attack.mitre.org/techniques/T1110/`. The project does not call the
MITRE website, external MITRE APIs, external threat intelligence, or external AI
services.

## Phase 6 Explanations and Triage Playbooks

`config/triage_steps.yaml` contains local defensive playbook steps. The selector
first looks for event-type and severity matches, then event-type fallbacks, then
a documented global fallback step.

Explanations are generated by a deterministic template engine from validated
alerts, classification results, local MITRE mappings, and selected playbook
steps. The engine does not use LLM calls, external AI APIs, API keys, stochastic
generation, network calls, or external enrichment.

CLI explanation output omits `raw_message` by default and sanitizes approved fake
sensitive markers before returning summaries, reasoning, evidence points, and
triage recommendations.

## Phase 7 Incident Grouping

`config/grouping_rules.yaml` contains local, deterministic correlation rules.
The grouping engine applies enabled rules, creates alert-to-alert correlation
edges, merges connected groups, and emits stable incident IDs such as
`INC-0001`.

Implemented grouping rules include:

- Same synthetic source IP within 15 minutes.
- Same synthetic username within 60 minutes.
- Same safe hostname when any related alert is `CRITICAL`.
- Same local MITRE tactic within 30 minutes.
- Authentication-to-privilege-escalation or lateral-movement chains.
- Multi-stage activity on the same safe lab hostname.

Grouping output is explainable through correlation reason records. It does not
include `raw_message`, real credentials, API keys, external lookups, live
network calls, or external AI behavior.

## Phase 8 Redaction and Safe Serialization

The default redaction policy is code-defined as `default-report-safe-v1`; it is
not an executable config file. The serializer removes `raw_message` fields by
default, redacts `source_ip`, `dest_ip`, and `username` values, redacts approved
fake marker constants, and redacts credential-looking patterns such as
`password=`, `token=`, `secret=`, `api_key=`, and `private_key`.

Safe metadata remains available for CLI output and reports: alert IDs, event
types, severities, MITRE technique IDs/names, tactics, template IDs, playbook
IDs, incident IDs, correlation rule IDs, timestamps, and safe `.test` hostnames.

The redaction layer remains deterministic, offline-only, and local. It makes no
network calls and uses no external AI APIs, LLM calls, or API keys.

## Phase 9 Reporting

Reports use the existing local YAML configuration indirectly through the
pipeline stages. The `report` command supports optional local config flags for
rules, MITRE mappings, triage playbooks, and grouping rules.

Report output paths must be local directories. URLs, network paths, and
parent-directory traversal are rejected. The report writer creates the local
output directory when needed and writes stable JSON/Markdown filenames.
